# Security Policy

Checkrd sits between AI agents and every external API they call. A
vulnerability here can silently bypass customer policy or leak
signed-telemetry credentials, so security reports are prioritized
over feature work.

Related documents:

- [THREAT-MODEL.md](./THREAT-MODEL.md) — assets, actors, mitigations.
- [WASM-CORE.md](./WASM-CORE.md) — the embedded WASM binary's I/O
  contract, cryptographic primitives, and integrity verification
  recipe.

## Reporting a vulnerability

**Do not open a public GitHub issue for a security vulnerability.**

Email: `security@checkrd.io`

For encrypted reports, fetch our PGP key at
`https://checkrd.io/.well-known/security.asc`. A machine-readable
`security.txt` (RFC 9116) is published at
`https://checkrd.io/.well-known/security.txt`.

Include in the report:

1. A description of the issue and the affected component (the
   `checkrd` package version and, if from source, the commit SHA).
2. Steps to reproduce, ideally with a minimal proof of concept.
3. Your assessment of impact (what can an attacker do?).
4. Your name and affiliation, if you want attribution in the
   advisory.

## Response commitment

- **Acknowledgement within 2 business days.** You receive a case
  ID.
- **Triage within 5 business days.** Severity class (none / low /
  medium / high / critical) and the expected fix window are
  confirmed.
- **Fix windows.** Critical: shipped within 7 days. High: within 30
  days. Medium and low: next minor release.
- **Coordinated disclosure.** A CVE and a GitHub Security Advisory
  are published when a fix is available. Reporter credit is
  included unless you request otherwise. Default embargo is 72
  hours from CVE issuance; longer if patch complexity requires it.

## Scope

In scope:

- The `checkrd` npm package.
- The bundled WASM core (`checkrd_core.wasm`) and its SHA-256
  integrity verification.
- The control-plane wire protocol as exercised by this SDK:
  `/v1/telemetry`, `/v1/agents/.../control`,
  `/v1/agents/.../public-key`.
- Signing, key handling, and SSE dispatch in `src/engine.ts`,
  `src/batcher.ts`, `src/receiver.ts`, `src/control.ts`.
- The vendor instrumentors under `src/integrations/` and the fetch
  interception layer in `src/transports/fetch.ts`.

Out of scope (report to the control-plane repository's
`SECURITY.md`):

- Server-side policy evaluation, authentication, and rate-limiting
  in the hosted control plane.
- The dashboard UI.

See [THREAT-MODEL.md § Out of Scope](./THREAT-MODEL.md#out-of-scope)
for the full taxonomy.

Security fixes are backported to the current major and the previous
major (N-1) for 12 months from the new major's release.

## Fail-closed defaults

The SDK ships with two fail-closed defaults. Do not weaken them in
production.

1. **`securityMode: "strict"`** (default). If the WASM engine fails
   to load with a user-supplied policy, `CheckrdInitError` is
   thrown. Setting `securityMode: "permissive"` (or
   `CHECKRD_SECURITY_MODE=permissive`) degrades to pass-through and
   is intended only for controlled rollouts.
2. **1 MB body-inspection limit.** Requests with bodies above 1 MB
   pass `null` into the policy engine for body matchers. Hosts in
   `securityMode: "strict"` SHOULD reject such requests rather than
   relying on name/URL matchers alone.

### Developer-only environment variables

- `CHECKRD_SKIP_WASM_INTEGRITY=1` — skips the WASM SHA-256 check.
  Intended for source checkouts where the hash file is not
  regenerated. Never set in production.

## Supply chain

- npm releases are published with
  [npm provenance](https://docs.npmjs.com/generating-provenance-statements)
  via GitHub Actions OIDC. Long-lived tokens are not used.
- Release tarballs are additionally signed with
  [Sigstore](https://sigstore.dev/); attestations are published on
  each GitHub Release.
- The bundled `checkrd_core.wasm` is verified at engine
  construction against a SHA-256 recorded at build time
  (`src/_wasm_integrity.ts::EXPECTED_SHA256`). Independent
  verification against the Sigstore attestation is documented in
  [WASM-CORE.md § Integrity Verification](./WASM-CORE.md#integrity-verification).
- A CycloneDX SBOM is generated by `npm run sbom` and attached to
  each GitHub Release. The SBOM includes a Cryptography Bill of
  Materials (CBOM) entry for Ed25519.
- Runtime dependencies are pinned in `package.json`;
  `package-lock.json` is committed. The release process runs
  `npm audit --omit=dev --audit-level=high` and `publint` on every
  publish.

## Telemetry signing

Every telemetry batch is signed before leaving the process using
IETF-standard primitives:

- **Ed25519** ([RFC 8032](https://www.rfc-editor.org/rfc/rfc8032.html))
  via `ed25519-dalek`, tested against RFC 8032 §7.1 vectors and the
  full [Project Wycheproof](https://github.com/google/wycheproof)
  Ed25519 corpus (150 vectors, 0 failures).
- **HTTP Message Signatures** ([RFC 9421](https://www.rfc-editor.org/rfc/rfc9421.html)),
  verified byte-for-byte against the §B.2.6 worked example.
- **Content-Digest** ([RFC 9530](https://www.rfc-editor.org/rfc/rfc9530.html))
  binding the request body into the signature.
- **Replay protection** via `created` + `expires` parameters with a
  5-minute signing window.

Mutation-tested to 100% kill rate on the verification primitives
(`cargo-mutants`). Cross-interop against the Python SDK's signing
path is gated on every release.

## Browser-runtime posture

This SDK is **not** intended to run in a browser. The control-plane
API key and the Ed25519 private key live in host process memory; a
browser bundle would ship both to end users. If a future release
adds browser support, it will require an explicit
`dangerouslyAllowBrowser: true` opt-in, following the pattern used
by the OpenAI and Anthropic SDKs. Until then, using this package in
a browser build is unsupported and a likely misconfiguration — file
an issue rather than working around it.

## Known limitations

- **No external cryptographic audit as of this release.** First
  audit scheduled for post-1.0; scope will be named here on
  confirmation.
- **No FIPS 140-3 validation.** Contact `security@checkrd.io` for
  the roadmap.
- **Side-channel timing is best-effort** under WebAssembly JITs
  (V8, JavaScriptCore, SpiderMonkey). See
  [WASM-CORE.md § Side-Channel Posture](./WASM-CORE.md#side-channel-posture).
## Signed policy-bundle install

The `policy_updated` SSE event is wired through `receiver.ts`'s
`policyUpdate` option (defaulted to the bundled trust list at
`src/_trust.ts::trustedPolicyKeysJson`) and dispatched via
`control.ts::installSignedPolicy` to the WASM core's
`reloadPolicySigned`. Every update goes through the full verification
path:

- DSSE signature verification against the trusted key list.
- Monotonic version check (rollback rejection).
- Bundle freshness check (default 24 h).
- Cross-type replay defense via DSSE payload-type binding.

On any failure the previous policy is left in place and a
`PolicySignatureError` is logged with the stable `code` /
`ffiCode` for dashboard grouping. Set `policyUpdate: null` on
`ControlReceiver` to disable the install path — intended for
debugging only.
